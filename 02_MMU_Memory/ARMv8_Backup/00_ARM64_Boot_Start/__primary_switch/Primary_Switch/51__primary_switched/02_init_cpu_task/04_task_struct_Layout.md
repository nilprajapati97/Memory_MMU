# `task_struct` Layout — What `init_cpu_task` Reads and Writes

## `task_struct` and `init_task` — The Big Picture

`init_task` is the boot CPU's task — PID 0, the "swapper" process. It is statically
allocated at compile time in `kernel/init_task.c`. `init_cpu_task` reads specific
fields from it and uses them to configure CPU hardware registers.

---

## `task_struct` Memory Layout (Fields Used by `init_cpu_task`)

```c
struct task_struct {
/* [offset 0]   */ struct thread_info thread_info;   // EMBEDDED, not a pointer
/* [thread_info size ≈ 136 bytes] */
                    ...
/* [TSK_STACK]  */ void *stack;     // pointer to init_stack (bottom of 16KB)
                    ...
/* many other fields */
};
```

### `thread_info` — Embedded at the START of `task_struct`

```c
struct thread_info {
/* [+0]  */ unsigned long flags;        // TIF_* flags (sigpending, need_resched, ...)
/* [+8]  */ u64  ttbr0;                // user page table for kpti
/* [+16] */ union {
                u64 preempt_count;      // preemption depth counter
                struct { u8 ...; };
            };
/* [+24] */ u32  cpu;                  // TSK_TI_CPU — current CPU ID ← OP5 reads this
/* [+28] */ u32  _unused;
/* [+32] */ u64  __reserved_for_virt;  // virtualization scratch
            ...
#ifdef CONFIG_SHADOW_CALL_STACK
            void *scs_base;             // SCS region base address
            void *scs_sp;              // TSK_TI_SCS_SP — SCS stack pointer ← OP4 reads this
#endif
};
```

## Exact Offsets (Generated by `asm-offsets.c`)

These are NOT magic numbers — they are GENERATED at build time:

```c
// arch/arm64/kernel/asm-offsets.c
DEFINE(TSK_STACK,           offsetof(struct task_struct, stack));
DEFINE(TSK_TI_CPU,          offsetof(struct task_struct, thread_info.cpu));
DEFINE(TSK_TI_SCS_SP,       offsetof(struct task_struct, thread_info.scs_sp));

DEFINE(THREAD_SIZE,         THREAD_SIZE);      // 16384
DEFINE(S_STACKFRAME,        offsetof(struct pt_regs, stackframe));
DEFINE(S_STACKFRAME_TYPE,   offsetof(struct pt_regs, stackframe) +
                             offsetof(struct stackframe_record, type));
DEFINE(PT_REGS_SIZE,        sizeof(struct pt_regs));
```

**Output file:** `arch/arm64/include/generated/asm-offsets.h`
```c
// Generated — do not edit
#define TSK_STACK           24       // example value
#define TSK_TI_CPU          24       // within thread_info
#define TSK_TI_SCS_SP       64       // example
#define THREAD_SIZE         16384
#define S_STACKFRAME        288      // within pt_regs
#define PT_REGS_SIZE        336
```

## `init_task` Static Initialization

```c
// kernel/init_task.c
struct task_struct init_task
__init_task_data = {
    /* task_struct fields: */
    .state          = 0,
    .stack          = init_stack,        // ← TSK_STACK field
    .active_mm      = &init_mm,
    .pid            = 0,
    .tgid           = 0,
    .comm           = INIT_TASK_COMM,    // "swapper"
    .flags          = PF_NOFREEZE,

    /* thread_info (embedded): */
    .thread_info = INIT_THREAD_INFO(init_task),
};

// INIT_THREAD_INFO macro:
#define INIT_THREAD_INFO(tsk) {
    .flags      = 0,
    .cpu        = 0,       // ← TSK_TI_CPU = 0 (boot CPU)
}
```

## `init_stack` — The 16KB Boot Stack

```c
// arch/arm64/include/asm/thread_info.h
#define INIT_TASK_DATA(align)   \
    ...                          \
    INIT_TASK_DATA_SECTION(align)

// arch/arm64/kernel/vmlinux.lds.S
INIT_TASK_DATA_SECTION(THREAD_SIZE)  // ensures 16KB alignment

// kernel/init_task.c
unsigned long init_stack[THREAD_SIZE / sizeof(unsigned long)]
    __initdata_aligned(THREAD_SIZE);
```

`init_stack` is in the `.init.data` section (freed after kernel init completes).
It is aligned to `THREAD_SIZE` (16KB) — required by the stack overflow detection
mechanism which uses `(sp & ~(THREAD_SIZE-1))` to find `thread_info` at the base.

## What `init_cpu_task` Reads vs Writes

### READS from `task_struct` (`init_task`):

| Field | Offset | Instruction | Value |
|---|---|---|---|
| `stack` | `TSK_STACK` | `ldr x5, [x4, #TSK_STACK]` | `&init_stack[0]` |
| `thread_info.cpu` | `TSK_TI_CPU` | `ldr w6, [x4, #TSK_TI_CPU]` | `0` (boot CPU) |
| `thread_info.scs_sp` | `TSK_TI_SCS_SP` | `ldr x18, [x18, #TSK_TI_SCS_SP]` | SCS pointer |

### WRITES to CPU registers (not to memory):

| Register | Instruction | Value Written |
|---|---|---|
| `sp_el0` | `msr sp_el0, x4` | `&init_task` |
| `SP` | `add/sub sp` | `init_stack top - PT_REGS_SIZE` |
| `x29` | `add x29, sp, #S_STACKFRAME` | frame pointer into pt_regs |
| `x18` | `scs_load_current` | `init_task.thread_info.scs_sp` |
| `tpidr_el1` | `set_this_cpu_offset` | `__per_cpu_offset[0]` |

### WRITES to memory (into init_stack):

| Memory Location | Instruction | Value |
|---|---|---|
| `[sp + S_STACKFRAME]` | `stp xzr, xzr` | fp=0, lr=0 |
| `[sp + S_STACKFRAME_TYPE]` | `str tmp1` | `FRAME_META_TYPE_FINAL` |

## Memory Access Pattern — Cache Line Analysis

`init_task` is in `.data` section, a fixed virtual address. At the time `init_cpu_task`
runs, the D-cache is ON and `init_task` is likely already in cache (it was accessed
during `__pi_early_map_kernel` for relocation purposes).

`init_stack` is in `.bss`, zero-initialized at load time by the bootloader or
`__pi_early_map_kernel`. At entry to `__primary_switched`, its cache lines may be
cold. The `stp xzr, xzr` and `str` instructions will perform write-allocate on the
L1 D-cache — bringing the cache line in on first write.

## ARMv8 CPU / Kernel / Memory Context

### CPU Perspective (ARMv8-A)
In ARMv8-A, the current task (process) is identified at EL0 via TPIDR_EL0 (user thread ID) and at EL1 via SP_EL0. Linux uses SP_EL0 to store the pointer to the current task_struct. SP_EL0 is a dedicated register (not the EL0 stack pointer when running in EL1 -- at EL1, SP_ELx selects either SP_EL0 or SP_EL1 as the active stack, controlled by PSTATE.SP). When the kernel uses SP_EL0 to store the current task pointer, it is using SP_EL0 as a general-purpose register (reading/writing it with MRS/MSR SP_EL0).

### Kernel Perspective (Linux ARM64)
init_cpu_task is a per-CPU variable (or boot-time initialization) that sets up the idle task (init_task / swapper) as the current task. In __primary_switched:
  msr  sp_el0, x23        // x23 holds init_task VA, set SP_EL0 = &init_task
  ldr  x8, [x23, #TSK_TI_CPU]  // verify .cpu field
The current macro in Linux ARM64 expands to:
  mrs x0, sp_el0          // read SP_EL0 as current task_struct pointer
SP_EL0 is never spilled to the stack (it is a system register), making current() essentially a zero-cost operation.

### Memory Perspective (ARMv8 Memory Model)
task_struct for init_task lives in the .data section of the kernel image (statically allocated). Its VA is in the kernel text/data mapping (TTBR1_EL1). When SP_EL0 is set to &init_task, the memory region is already mapped and accessible. The task's stack (thread_union) is in the .init.data section and is also already mapped. After start_kernel -> sched_init(), all subsequent tasks have their task_struct allocated from slab memory in the kernel heap (also in the TTBR1_EL1 region).
