# `S_STACKFRAME` Offset — Where the Sentinel Lives in `pt_regs`

## Finding `S_STACKFRAME` in the Build System

`S_STACKFRAME` is NOT a hardcoded number in assembly code. It is generated by the
kernel build system from C struct definitions:

```c
// arch/arm64/kernel/asm-offsets.c

// This file is compiled to an object, then its
// assembly listing is parsed to extract DEFINE() values

DEFINE(S_STACKFRAME,
       offsetof(struct pt_regs, stackframe));
DEFINE(S_STACKFRAME_TYPE,
       offsetof(struct pt_regs, stackframe) +
       offsetof(struct frame_meta, type));
```

**Build process:**
```
1. gcc -S asm-offsets.c → asm-offsets.s (assembly listing)
2. scripts/extract-offsets < asm-offsets.s → asm-offsets.h
3. asm-offsets.h #included by head.S and other .S files
```

This guarantees that if the C struct changes (a field is added/reordered), the
assembly code AUTOMATICALLY uses the new offsets. No manual synchronization needed.

## `struct pt_regs` Layout at the `stackframe` Field

```c
struct pt_regs {
    // [0..271]:   user registers (x0-x30, sp, pc, pstate) = 272 bytes
    union {
        struct user_pt_regs user_regs;
        struct {
            u64 regs[31];  // x0-x30
            u64 sp;
            u64 pc;
            u64 pstate;
        };
    };                      // = 272 bytes at offset 0

    // [272..279]: orig_x0
    u64 orig_x0;            // +272, 8 bytes

    // [280..287]: syscallno + padding
    s32 syscallno;          // +280, 4 bytes
    u32 unused2;            // +284, 4 bytes

    // [288..295]: sdei_ttbr1
    u64 sdei_ttbr1;         // +288, 8 bytes

    // [296..303]: pmr_save
    u64 pmr_save;           // +296, 8 bytes

    // [304..319]: stackframe (fp + pc pair)   ← S_STACKFRAME = 304
    u64 stackframe[2];      // +304, 16 bytes

    // [320..327]: lockdep_hardirqs
    u64 lockdep_hardirqs;   // +320, 8 bytes

    // [328..335]: exit_rcu
    u64 exit_rcu;           // +328, 8 bytes

    // Total: 336 bytes = PT_REGS_SIZE
};
```

So:
- `S_STACKFRAME = 304`
- `S_STACKFRAME_TYPE = 304 + 16 = 320` (if type field is after the [2] array)

Note: The exact layout depends on kernel version and config. The DEFINE macros
in `asm-offsets.c` always give the correct values for the current build.

## The Three Memory Writes

```asm
// Write 1: fp=0, lr=0
stp     xzr, xzr, [sp, #S_STACKFRAME]
//  [sp+304] = 0  (stackframe[0] = fp = 0)
//  [sp+312] = 0  (stackframe[1] = lr/pc = 0)

// Write 2: type = FINAL
mov     tmp1, #FRAME_META_TYPE_FINAL    // tmp1 = 1
str     tmp1, [sp, #S_STACKFRAME_TYPE]
//  [sp+320] = 1  (stackframe_type = FINAL)

// Write 3: update x29 to point to the stackframe
add     x29, sp, #S_STACKFRAME
//  x29 = sp + 304
```

---

## Why `str` Not `strb` or `strh` for Type?

```asm
str     tmp1, [sp, #S_STACKFRAME_TYPE]
```

`str` (without suffix) stores a 64-bit value. But `FRAME_META_TYPE_FINAL = 1`
only needs 1 byte. Why not `strb` (store byte)?

The `stackframe_type` field might be defined as `u64` in the `pt_regs` structure
for alignment reasons. If it's `u64`, a `strb` would only write to 1 byte and
leave the other 7 bytes as their previous value (zeros from `init_stack` BSS,
but not explicitly set by this code).

The `str` (64-bit store) writes all 8 bytes atomically with the value 1. This
is both more correct (writes the full field) and potentially faster (aligned
8-byte writes are optimal for most CPU memory subsystems).

## Verifying the Values — Using `crash` or `gdb`

To verify the frame sentinel after boot:

```bash
# Using crash utility on a live kernel:
crash /proc/kcore /boot/vmlinux
crash> x/4gx init_task.stack+THREAD_SIZE-PT_REGS_SIZE+S_STACKFRAME
0xffff800010xxxxxx: 0x0000000000000000  0x0000000000000000
0xffff800010xxxxxx: 0x0000000000000001  0x0000000000000000

# First two 8-byte values: fp=0, lr=0
# Third 8-byte value: type=1 (FRAME_META_TYPE_FINAL)
```

This confirms `init_cpu_task` correctly initialized the frame sentinel.

## Frame Sentinel Across CPU Hotplug

When a secondary CPU is brought online, `init_cpu_task` also writes a frame sentinel
into that CPU's idle task stack. The process:

1. `cpu_up(N)` allocates idle task for CPU N
2. `__cpu_up` calls architecture code to start CPU N
3. CPU N starts at `secondary_entry` (head.S)
4. `secondary_switched` calls `init_cpu_task` with CPU N's idle task
5. Frame sentinel written to CPU N's idle stack

Each CPU's idle task has its own frame sentinel. Stack traces from CPU N's idle task
terminate cleanly at its own sentinel.

## ARMv8 CPU / Kernel / Memory Context

### CPU Perspective (ARMv8-A)
In ARMv8-A, the stack pointer is a dedicated register (SP_EL1 at EL1, SP_EL0 at EL0). SP_EL1 is the stack pointer used by the kernel during normal execution. The AAPCS64 ABI requires the stack to be 16-byte aligned at any instruction that may cause an exception. SCTLR_EL1.SA (bit 3) enables hardware enforcement of this alignment: if SP_EL1 is not 16-byte aligned when a load/store using SP is executed, an SP alignment fault is raised. The frame pointer (x29) is a general-purpose register used by convention to hold the base of the current stack frame. Writing x29 is the first act of any C function that wishes to be unwound.

### Kernel Perspective (Linux ARM64)
After the MMU is enabled, __primary_switch reinitializes the stack pointer to a virtual address. The early boot stack is defined as:
  __INIT_DATA: init_thread_union (size THREAD_SIZE, typically 16 KB)
The LDR instruction loads the VA of init_thread_union + THREAD_SIZE into x0, then MOV sp, x0 sets SP_EL1. This is necessary because the old stack pointer was set to a physical address (before the MMU) and that PA is no longer the correct address for the kernel VA layout. x29 is set to zero (zero frame pointer) to terminate the unwind chain at the first kernel stack frame.

### Memory Perspective (ARMv8 Memory Model)
The kernel stack resides in Normal Inner-Shareable Write-Back Cacheable memory (MT_NORMAL). Once the MMU and D-cache are enabled, all stack accesses (PUSH/POP equivalents: STP/LDP) go through the L1 D-cache. The L1 D-cache write-back policy means that the stack contents are not immediately visible to physical memory until a cache clean or eviction. This is safe for the stack because the kernel does not use DMA to read stack memory. The stack pointer reinitalization at VA is a hard cut: all future kernel stack frames exist in the high VA kernel mapping.
